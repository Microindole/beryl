# 协作规则补充

- 本语言设计哲学是简洁和规范，不追求 C 式底层细节，也避免 Java 式冗长臃肿。
- 本语言要求静态类型，语义结果要清晰明了；允许多态，但必须可诊断、可解释。
- 开发时遵循开闭原则，避免单文件职责过重，避免无边界扩张的 if-else 分派。
- 不要轻易修改已有稳定代码；改动要优先局部化，避免无关回归。
- 不要在没有明确需求、已知失败路径或可复现风险的情况下添加“预防性”错误处理、兜底分支或回退逻辑。
- 不要给显而易见的代码写解释性废话注释；注释只用于说明意图、约束、边界条件或非直观设计。
- 发现重复逻辑时先评估提取公共实现的收益；若重复已超过两处且后续明显还会扩展，不得继续复制粘贴。
- 若新增实现导致代码量、分支数、文件数或抽象层级明显增长，而用户可见收益很小，必须重新审查设计，重点检查重复逻辑、硬编码、过度防御、无效抽象和胶水代码。
- 采用“实现 / 收敛”交替节奏：
  - Session A：实现目标功能。
  - Session B：禁止新增功能，只做去重、去硬编码、提取常量、删除无效分支、压缩样板代码与补测试。
  - 目标是降低冗余并提升可维护性，而不是机械追求固定降行数指标。
- 任何行为变更、缺陷修复或重构，只要影响可观察行为、语义边界或实际代码路径，必须补充或更新对应测试；优先使用 `tests/integration/` 或 `tests/example/` 下的回归用例。

## 重构分层规则（非强制，默认优先）

- 目标：减少同层目录污染，保持模块可发现性与可维护性。
- 默认策略：优先“下沉子目录”而不是“继续在同层新增文件”。
- 触发建议（任一命中即可考虑下沉）：
  - 同层文件数接近或超过 9（启发式预警线，可按目录复杂度调整，不是工业硬标准）；
  - 单文件超过 300 行且还要继续扩展；
  - 同一主题出现 3 个以上并列文件（例如 `expr_*`、`test_steps_*`）。
- 命名建议：
  - 业务层：`domain/subdomain/*.lcy|rs`，例如 `resolver/expr/call.lcy`；
  - 测试层：`driver/steps/*.lcy` 或 `integration/<topic>/*.lcy`。
- 例外：
  - 小型目录（短期稳定、文件很少）可暂不下沉；
  - 一次性迁移窗口可先平铺落地，但必须在后续提交补齐下沉整理。

## 文档同步规则

- `prompt/context.md`：实时记录每轮实现与验证结果（默认必更新）。
- `prompt/sprint/status.md`：只记录阶段性里程碑、风险收口与发布级状态，不做每次小改动流水账。
- 若两者出现冲突，以“`context.md` 实时、`status.md` 汇总”的职责划分为准，必须先修正文档冲突再继续开发。
